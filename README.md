## ТЗ
Нижняя часть экрана - горизонтальный скролл из 20 прямоугольников (далее - кубики) разных
цветов.
Любой из кубиков можно перенести в правую верхнюю часть и поставить друг на друга в виде
башни с условиями:
- первый кубик можно поставить в любое место правой верхней части экрана;
- остальные кубики ставятся в башню только если их перенесли поверх уже выставленных, в этом
случае они анимацией подпрыгивают и ставятся сверху башни;
- если игрок перенес кубик и промазал мимо башни, кубик с анимацией исчезает или взрывается;
- кубики ставятся поверх предыдущих в башню со случайным смещением по горизонтали, но не
более 50% длины грани;
- ограничением по высоте является высота экрана устройства, если последний кубик вышел за ее
границу, то больше надстраивать башню нельзя;
- кубики внизу бесконечные и не пропадают после перетаскивания наверх,
- физика не используется, только анимации.
В любой момент кубик из башни можно выкинуть в дыру, перенеся его на ее изображение в левой
верхней части экрана. В этом случае, те что были выше башни плавно опускаются вниз анимацией.
Необходимо учесть ее овальные границы при оценке попал ли игрок в нее, перетаскивая кубик.
Под каждой частью экрана свой фон.
Каждое действие - установка кубика, выкидывание кубика, пропадание кубика, ограничение по
высоте - должно сопровождаться надписью над нижней частью экрана.
Конфигурация игры должна содержать: количество кубиков внизу и их цвета

Требования:
1. Код должен предусматривать масштабирование для будущих обновлений (к примеру нужно будет
переносить новый кубик только на кубик такого же цвета в башне);
2. Предусмотрите возможность локализации (можно без самой системы);
3. Нужно учесть что источником конфигурации игры могут стать разные источники данных (в игре
может быть 1 реализация - из ScriptableObject);
4. Drag & drop кубиков должен работать корректно с учетом скролла (не должно быть ситуаций,
когда при начале перетаскивания скроллируется нижняя полоса);
5. Необходимо чтобы мини-игра выглядела хорошо на основных соотношениях сторон мобильных
устройств: 19.5x9, 16x9, 4x3.
6. Должна использоваться как минимум 1 анимация (Unity Animation или DoTween);
7. Прогресс сохраняется между сессиями;

## Пояснение
Основная логика построена на поступлении данных о кубах в реактивне коллекции и их дальнейшей обработке.
Обработчики представлены в **ApplicationInstaller**.
* **DragAndDropHandler** обрабатывает реактивную коллекцию в которую помещаются кубы для добавления на них механики Drag&Drop
* **CubesScrollHandler** добавляет кубы из соответствующей реактивной коллекции в скролл на экране и добавляет механику появления Drag&Drop куба при нажатии на куб из скрола
* **HoleHandler** обрабатывает дроп куба в яму
* **TowerCollectionHandler** строит из кубов из соответствующей коллекции башенку в соответсвии с данными из словаря с позициями кубов
* **OnTowerDropHandler** обрабатывает дроп на экран с башней. Проверяет, можно ли дропнутый куб разместить на башне и, если можно, добавляет его в соответствующуу реактивную коллекцию
## Архитектурные решения
* **Паттерн "Команда"** используется для уменьшения связности кода и возможности переиспользования и кеширования. Особенно его удобно использовать в связке с DI, который может в любой момент прокинуть зависимости в команду
* **Паттерн "Декоратор"** используется для возможности добавить новый функционал без модификации старого кода. Используется при проверке возможности разместить куба на башню
